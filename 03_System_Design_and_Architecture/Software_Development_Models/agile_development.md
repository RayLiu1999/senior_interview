# 敏捷開發 (Agile Development)

- **難度**: 6
- **標籤**: `Agile`, `Software Development Models`, `Methodology`

## 問題詳述

什麼是敏捷開發 (Agile Development)？它的核心價值觀和十二項原則是什麼？它與傳統的瀑布模型有何不同？

## 核心理論與詳解

敏捷開發 (Agile Development) 不是一種具體的方法或流程，而是一套指導軟體開發的價值觀和原則。它強調以人為本、迭代、循序漸進的開發方式，旨在應對複雜和不確定的需求，並快速交付有價值的軟體。敏捷的核心思想是透過短週期的迭代（稱為 Sprint 或 Iteration）來頻繁地交付可工作的軟體，並在每個週期結束時收集回饋，以便在下一個週期中進行調整。

### 敏捷宣言的四個核心價值觀

敏捷開發的基礎是《敏捷軟體開發宣言》（Manifesto for Agile Software Development），它提出了四個核心價值觀：

1.  **個人與互動** 重於 流程與工具 (Individuals and interactions over processes and tools)
    -   強調團隊成員之間的溝通與協作，認為優秀的人才和高效的溝通是專案成功的關鍵，而非死板的流程和工具。

2.  **可工作的軟體** 重於 詳盡的文件 (Working software over comprehensive documentation)
    -   雖然文件是必要的，但敏捷更看重能實際運作並為客戶帶來價值的軟體。目標是交付功能，而不是堆砌文件。

3.  **客戶合作** 重於 合約協商 (Customer collaboration over contract negotiation)
    -   強調在整個開發過程中與客戶保持緊密的合作關係，將客戶視為團隊的一部分，共同探索需求和解決方案，而不是在專案初期就試圖用合約鎖定所有細節。

4.  **回應變化** 重於 遵循計畫 (Responding to change over following a plan)
    -   承認在軟體開發中，變化是不可避免的。敏捷流程擁抱變化，並具備快速調整方向以應對新需求或市場變化的能力，而不是僵化地遵循一個可能已經過時的初始計畫。

### 敏捷開發的十二項原則

除了四個價值觀，敏捷宣言還闡述了十二項支持這些價值觀的原則，它們為如何實踐敏捷提供了更具體的指導：

1.  我們最優先的任務，是透過及早並持續地交付有價值的軟體來滿足客戶。
2.  歡迎需求的改變，即使是在開發的晚期。敏捷流程駕馭變動，以維護客戶的競爭優勢。
3.  經常交付可工作的軟體，頻率可以從數週到數個月，以較短的週期為佳。
4.  業務人員與開發者必須在專案全程中天天一起工作。
5.  以積極的個人來建構專案，給予他們所需的環境與支援，並信任他們可以完成工作。
6.  在團隊中及團隊之間，最有效率且有效能的資訊傳遞方式是面對面溝通。
7.  可工作的軟體是進度的首要衡量標準。
8.  敏捷流程提倡可持續的開發。贊助者、開發者及使用者應當能不斷地維持穩定的步調。
9.  持續追求技術的卓越及良好的設計，以強化敏捷性。
10. 精簡──盡量減少非必要工作的藝術──是不可或缺的。
11. 最好的架構、需求與設計皆來自於能自我組織的團隊。
12. 團隊定期自省如何更有效率，並據此調整與改善自身的行為。

### 與瀑布模型的比較

| 特性 | 瀑布模型 (Waterfall) | 敏捷開發 (Agile) |
| :--- | :--- | :--- |
| **流程** | 線性、順序性（需求 -> 設計 -> 開發 -> 測試 -> 部署） | 迭代、增量式 |
| **週期** | 長，通常以月或年為單位 | 短，通常為 1-4 週 |
| **需求** | 在專案初期就完全確定並凍結 | 歡迎並適應整個過程中的需求變化 |
| **交付** | 在專案結束時一次性交付所有功能 | 頻繁、持續地交付可工作的軟體增量 |
| **回饋** | 在專案後期才能獲得客戶回饋 | 在每個迭代結束時都能獲得回饋 |
| **風險** | 風險集中在專案後期，問題可能很晚才被發現 | 風險分散在每個迭代中，可以及早發現並應對問題 |
| **文件** | 強調詳盡的規格和設計文件 | 強調可工作的軟體，文件求精簡實用 |
| **客戶參與** | 主要在專案初期和結束時參與 | 在整個開發過程中持續合作 |

## 程式碼範例 (可選)

軟體開發模型是一種思想框架，無法直接用程式碼表示。但我們可以透過一個簡單的註解來比喻其開發過程。

```go
// 這個範例旨在比喻開發模型的流程，而非實際程式碼

package main

import "fmt"

// 瀑布模型：一步一步，完成後才能到下一步
func waterfallDevelopment() {
    fmt.Println("1. 完成所有需求分析...")
    fmt.Println("2. 完成所有系統設計...")
    fmt.Println("3. 完成所有程式碼開發...")
    fmt.Println("4. 完成所有軟體測試...")
    fmt.Println("5. 交付產品！")
}

// 敏捷開發：在短週期內完成一個小功能閉環
func agileIteration(sprintNumber int) {
    fmt.Printf("\n--- 第 %d 次迭代開始 ---\n", sprintNumber)
    fmt.Println("1. 規劃本次迭代要開發的小功能...")
    fmt.Println("2. 設計這個小功能...")
    fmt.Println("3. 開發這個小功能...")
    fmt.Println("4. 測試這個小功能...")
    fmt.Println("5. 交付這個可工作的小功能，並收集回饋！")
}

func main() {
    fmt.Println("=== 瀑布模型開發流程 ===")
    waterfallDevelopment()

    fmt.Println("\n=== 敏捷開發流程 ===")
    agileIteration(1)
    agileIteration(2)
    agileIteration(3)
}
```
